## How to Contribute

### 1. Create a New Agent

Have an idea for a specialized agent? Here's how to add one:

<Steps>
  <Step title="Fork the repository">
    ```bash
    # Fork on GitHub, then clone your fork
    git clone https://github.com/YOUR_USERNAME/agency-agents.git
    cd agency-agents
    ```
  </Step>
  
  <Step title="Choose the appropriate category">
    Select or propose a category:
    - `engineering/` - Software development specialists
    - `design/` - UX/UI and creative specialists
    - `marketing/` - Growth and marketing specialists
    - `product/` - Product management specialists
    - `project-management/` - PM and coordination specialists
    - `testing/` - QA and testing specialists
    - `support/` - Operations and support specialists
    - `spatial-computing/` - AR/VR/XR specialists
    - `specialized/` - Unique specialists that don't fit elsewhere
  </Step>
  
  <Step title="Create your agent file">
    ```bash
    # Create new agent file
    touch category/category-agent-name.md
    
    # Example:
    touch marketing/marketing-email-specialist.md
    ```
    
    Follow the [agent template](/contributing/agent-template) structure.
  </Step>
  
  <Step title="Test your agent">
    Use the agent in real scenarios:
    - Does it produce the expected outputs?
    - Is the personality distinctive?
    - Are the deliverables concrete and useful?
    - Do the workflows make sense?
  </Step>
  
  <Step title="Submit a Pull Request">
    ```bash
    # Create a branch
    git checkout -b add-agent-name
    
    # Add your changes
    git add .
    git commit -m "Add [Agent Name] specialist"
    
    # Push to your fork
    git push origin add-agent-name
    ```
    
    Then open a Pull Request on GitHub.
  </Step>
</Steps>

### 2. Improve Existing Agents

    2. Identified friction points (5-step process)
    3. Redesigned as 2-step with auto-save
    4. Implemented with progressive disclosure
    5. Result: 82% completion (+17%)

  <Tab title="Update Workflows">
    **What to Update**:
    - Modern best practices

### 4. Report Issues

Found a problem? Let us know:

<Steps>
  <Step title="Check existing issues">
    Search [GitHub Issues](https://github.com/msitarzewski/agency-agents/issues) to avoid duplicates
  </Step>
  
  <Step title="Provide context">
    Include:
    - What agent(s) are affected
    - What you expected to happen
    - What actually happened
    - Steps to reproduce
    - Your environment (if relevant)
  </Step>
  
  <Step title="Suggest solutions">
    If you have ideas for fixes, share them!
  </Step>
</Steps>

## Pull Request Process

### Before Submitting

<Tabs>
  <Tab title="New Agent Checklist">
    - [ ] Follows agent template structure
    - [ ] Includes personality and voice
    - [ ] Has concrete code/template examples (2-3 minimum)
    - [ ] Defines success metrics
    - [ ] Includes step-by-step workflow
    - [ ] Proofread and formatted correctly
    - [ ] Tested in real scenarios
    - [ ] Fills a real gap (not duplicate)
  </Tab>
  
  <Tab title="Improvement Checklist">
    - [ ] Improvement adds clear value
    - [ ] Examples are runnable/practical
    - [ ] Code follows modern best practices
    - [ ] Metrics are current and accurate
    - [ ] Documentation is clear
    - [ ] No typos or formatting issues
  </Tab>
</Tabs>

### Submitting Your PR

<Steps>
  <Step title="Create descriptive PR">
    ```markdown
    ## Agent Information
    **Agent Name**: [Name]
    **Category**: [engineering/design/marketing/etc.]
    **Specialty**: [One-line description]
    
    ## Motivation
    [Why is this agent needed? What gap does it fill?]
    
    ## Testing
    [How have you tested this agent? Real-world use cases?]
    
    ## Checklist
    - [x] Follows agent template structure
    - [x] Includes personality and voice
    - [x] Has concrete examples
    - [x] Defines success metrics
    - [x] Tested in real scenarios
    ```
  </Step>
  
  <Step title="Community review">
    Other contributors may provide feedback. Address comments and make improvements.
  </Step>
  
  <Step title="Iteration">
    Work with reviewers to refine your contribution.
  </Step>
  
  <Step title="Approval & merge">
    Once approved, maintainers will merge your contribution!
  </Step>
</Steps>

## Style Guide

### Writing Style

<CardGroup cols={2}>
  <Card title="Be Specific">
    "Reduce page load by 60%" not "Make it faster"
  </Card>
  <Card title="Be Concrete">
    "Create React components with TypeScript" not "Build UIs"
  </Card>
  <Card title="Be Memorable">
    Give agents personality, not generic corporate speak
  </Card>
  <Card title="Be Practical">
    Include real code, not pseudo-code
  </Card>
</CardGroup>

### Formatting Guidelines

<Tabs>
  <Tab title="Markdown">
    ```markdown
    # Use proper heading hierarchy
    
    ## Section headers with emojis (optional)
    
    ### Subsections
    
    - Use **bold** for emphasis
    - Use `code` for technical terms
    - Use > blockquotes for important notes
    ```
  </Tab>
  
  <Tab title="Code Blocks">
    ```markdown
    \`\`\`typescript
    // Always include:
    // 1. Language specification
    // 2. Comments explaining key concepts
    // 3. Real, runnable code
    // 4. Modern best practices
    
    interface AgentExample {
      name: string;
      specialty: string;
    }
    \`\`\`
    ```
  </Tab>
  
  <Tab title="Tables">
    ```markdown
    | Column 1 | Column 2 | Column 3 |
    |----------|----------|----------|
    | Data     | Data     | Data     |
    
    Use for:
    - Comparing options
    - Showing metrics
    - Organizing structured data
    ```
  </Tab>
</Tabs>

### Tone Guidelines

<AccordionGroup>
  <Accordion title="Professional but Approachable">
    **Good**: "Let's optimize this API endpoint for better performance"
    
    **Avoid**: "We must optimize the API endpoint immediately" (too formal)
    
    **Avoid**: "Yo, this API is slow, let's fix it" (too casual)
  </Accordion>
  
  <Accordion title="Confident but Not Arrogant">
    **Good**: "Here's the best approach based on industry standards"
    
    **Avoid**: "Obviously, you should do this" (condescending)
    
    **Avoid**: "Maybe you could try this?" (too uncertain)
  </Accordion>
  
  <Accordion title="Helpful but Not Hand-Holding">
    **Good**: "Implement virtualization to handle large datasets efficiently"
    
    **Avoid**: "You need to implement virtualization because your table is slow" (assumes incompetence)
    
    **Avoid**: "First open your editor, then create a file..." (too basic)
  </Accordion>
</AccordionGroup>
